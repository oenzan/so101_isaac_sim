# Dual SO-101 in Isaac Sim — cloth folding workspace

Scripts to load the **bimanual SO-101 (SO-100) robot** into NVIDIA Isaac Sim and
set up a scene for **cloth folding** experiments.

For the full design rationale (build pipeline, motor model, camera mount, cloth),
see [`simulation_architecture.md`](simulation_architecture.md).

```
isaac_sim/
├── scripts/
│   ├── generate_urdf.py        # build a clean URDF from the arm description (plain python)
│   ├── config.py               # shared paths, joint names, poses, scene sizes, motor + camera params
│   ├── import_urdf_to_usd.py   # one-time: URDF -> USD via Isaac's importer
│   ├── motors.py               # Feetech STS3215 actuator model (gains, torque/speed limits)
│   ├── cameras.py              # wrist UVC cameras (TheRobotStudio 32x32 mount)
│   ├── cloth_utils.py          # PhysX particle-cloth helpers
│   ├── fk_check.py             # forward-kinematics pose checker (plain python)
│   └── setup_scene.py          # build + run the full scene (robot + cams + table + cloth)
├── urdf/
│   └── so_100_dual.urdf        # generated robot description (relative mesh paths)
├── usd/
│   └── so_100_dual.usd         # generated by import_urdf_to_usd.py
└── simulation_architecture.md  # how it all fits together
```

The robot is two identical 6-DOF arms welded to a shared `world` frame:
left base at `x = -0.3 m`, right base at `x = +0.3 m`. Each arm has
`Shoulder_Rotation, Shoulder_Pitch, Elbow, Wrist_Pitch, Wrist_Roll, Gripper`
(prefixed `left_` / `right_`) → **12 actuated joints** total.

## Requirements

- **NVIDIA Isaac Sim 4.5 / 5.0** (also works on 4.2 — the scripts auto-detect the
  `isaacsim.*` vs `omni.isaac.*` API namespaces).
- An **NVIDIA GPU**. Particle cloth runs on GPU PhysX, which `setup_scene.py`
  enables automatically.
- The STL meshes under `src/SO-100-arm/models/so_100_arm_5dof/meshes/`
  (already in this repo). The URDF references them with paths relative to itself,
  so keep the `isaac_sim/` folder next to `src/`.

Run every Isaac script with Isaac's bundled Python, **not** your system Python:

```bash
# Linux install dir example: ~/isaacsim  (use isaac-sim.sh's python.sh)
cd ~/isaacsim
./python.sh /full/path/to/so101_isaac_sim/isaac_sim/scripts/<script>.py
```

## Step 1 — (re)generate the URDF  *(optional; already generated)*

Plain Python, no Isaac needed. Run this if you change joint limits, the ready
pose, or the mesh location in `generate_urdf.py` / `config.py`.

```bash
python3 isaac_sim/scripts/generate_urdf.py
```

Produces `isaac_sim/urdf/so_100_dual.urdf` with **revolute joints + limits**
(the source description used limit-less `continuous` joints) and **relative mesh
paths** (the source used ROS `package://` paths that Isaac can't resolve).

## Step 2 — import URDF → USD  *(run once)*

```bash
./python.sh .../isaac_sim/scripts/import_urdf_to_usd.py
```

Writes `isaac_sim/usd/so_100_dual.usd`. Fixed joints are merged, the base is
anchored (table-mounted), and every revolute joint gets a **position drive** so
you can command joint targets. Re-run only when the URDF changes.

> **Re-import after the motor update:** the joint `effort`/`velocity` limits and
> default drive gains now reflect the **Feetech STS3215** servo, so **re-run
> Step 2** to bake them into the USD. (`motors.py` also re-applies the model at
> runtime, so an old USD still behaves, but a fresh import is cleaner.)

## Step 3 — build and run the scene

```bash
./python.sh .../isaac_sim/scripts/setup_scene.py                  # interactive viewport
./python.sh .../isaac_sim/scripts/setup_scene.py --headless       # no window
./python.sh .../isaac_sim/scripts/setup_scene.py --steps 600      # run 600 steps then exit
./python.sh .../isaac_sim/scripts/setup_scene.py --capture-cameras # also make readable Camera sensors
```

This assembles: ground plane, dome light, a table, the dual robot mounted on the
table top, **a wrist camera on each arm**, and a draped **particle-cloth** square
in front of the arms. The joints are given the **Feetech STS3215** motor model
(realistic torque/speed limits and control gains), then both arms are driven to a
"ready" pose and hold it. If the USD from Step 2 is missing, the scene imports the
URDF on the fly.

- **Motors** — `motors.py` applies the STS3215 model after load. Retune via the
  `MOTOR_*` / `*_DRIVE_*` constants in `config.py`.
- **Wrist cameras** — one per arm, on the `Fixed_Gripper` (wrist-roll) link,
  looking down the gripper. Select one as the viewport camera to see its view, or
  run with `--capture-cameras` and call `cam.sensor.get_rgba()` to read frames.
  Placement/FOV/resolution are the `WRIST_CAM_*` constants in `config.py`.

The script also enables the PhysX UI extensions so you can open
**Window → Physics → Physics Inspector** (and the Physics authoring toolbar) from
the menu — these are off by default in the minimal standalone app.

### Opening the Physics Inspector

Once the viewport is up: top menu **Window → Physics → Physics Inspector**.
(If you previously saw `Could not find action ... show_physics_inspector` in the
log, that was the `omni.physx.supportui` extension not being loaded — now enabled
automatically by `setup_scene.py`.)

The script applies the ready pose **once** and then only steps the simulation —
it does **not** re-command the joints every frame. That means when you scrub a
joint in the Physics Inspector it **stays** where you put it (earlier the joints
snapped back / vibrated because the script was overwriting the target every
frame). The ready pose is also stored as the articulation's default state, so a
GUI **Stop → Play** returns the arms to it.

### Log messages you can ignore (benign)

These appear on a normal run and do **not** indicate a problem:

- `Changing particle cloth mesh or PhysxSchemaPhysxAutoParticleClothAPI parameter
  during simulation is not supported` — fires once as the particle cloth cooks
  its rest state on the first sim step. The cloth still simulates correctly.
- `getAttributeCount/getTypes called on non-existent path .../node_STL_BINARY_`
  and `Unresolved reference prim path </World/so_100_dual/world/visuals>` — URDF
  importer artifacts from the empty `world` root link / mesh nodes. The robot
  renders and simulates fine. If they bother you, delete `isaac_sim/usd/` and
  re-run Step 2 for a clean import.
- `Disabling CCD for GPU dynamics` — expected: CCD isn't supported under GPU
  PhysX, which we enable for the cloth.

## How to drive the robot from your own code

`setup_scene.py:main()` shows the pattern. The articulation exposes the 12 DOFs
in `robot.dof_names`; command them with position targets:

```python
from isaacsim.core.utils.types import ArticulationAction  # omni.isaac.* on 4.2
targets = np.array([READY_POSE.get(n, 0.0) for n in robot.dof_names])
robot.get_articulation_controller().apply_action(
    ArticulationAction(joint_positions=targets)
)
```

Joint names, the default `READY_POSE`, and all scene dimensions live in
`config.py` — that's the file to edit when iterating.

### Choosing a pose without launching Isaac

`scripts/fk_check.py` is a tiny standalone forward-kinematics tool (plain
`numpy`). It prints the world position of every link for a candidate pose so you
can confirm the grippers clear the base/table *before* running the sim:

```bash
python3 isaac_sim/scripts/fk_check.py
```

The default `READY_POSE` keeps every joint at 0 (the collision-free CAD assembly)
and only swivels `Shoulder_Rotation` by 180° so both arms face the cloth. That is
why the grippers no longer fold into the base.

## Tuning notes for cloth folding

- **Cloth feel**: `cloth_utils.add_cloth(...)` exposes `spring_*_stiffness`
  (bend/shear/stretch) and `spring_damping`. Lower bend stiffness = floppier,
  easier-to-fold fabric. `CLOTH_RESOLUTION` in `config.py` trades fidelity for
  speed (40×40 is a good start; raise for finer folds).
- **Grasping cloth** is the hard part of a folding task. Particle cloth doesn't
  "stick" to a rigid gripper by default — for reliable pinch grasps you'll want
  a PhysX *particle attachment* between gripper jaw and the nearest cloth
  particles, or high-friction contact. This scene gives you the robot + cloth +
  physics foundation to build that grasp logic on top of.
- **Joint limits / drive gains**: `JOINT_LIMITS` in `generate_urdf.py` and the
  drive stiffness/damping in `import_urdf_to_usd.py` are conservative defaults;
  tune them to the real SO-101 servos (Feetech STS3215) as needed.
